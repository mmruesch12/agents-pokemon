Pokemon Gold Agent — Mental Model

How the autonomous player actually works today: one button press per step, rule-based early-game phases, optional LLM tie-breaking.

One sentence: Each step = read RAM → supervisor picks a specialist → press one button (or wait) → critic reviews → memory updates → stop (unless the runner asks for another step).

03 Supervisor routing (priority order)

First match wins. Most early game is rule-based — not free-form LLM exploration.

  1. Still bootstrapping (title / name / clock) bootstrap
  2. Active phase goal already complete idle
  3. In battle battler
  4. Script running, joypad blocked waiter
  5. Dialog or scripted interact needed interactor
  6. Mom scene / lab scene forced interactor
  7. Stuck too long or critic said replan planner
  8. Otherwise — walk somewhere navigator

05 How movement works (navigator)

Deterministic targets + pathfinding grids. LLM only picks among candidate directions.

1

Phase target

Active phase returns a fixed (x, y) for this map — e.g. stairs on 2F, lab warp in New Bark.

2

Pathfinding

find_path() on hand-tuned per-map grids (walkable tiles, blocked stairs during Mom scene).

3

Candidate directions

Path steps + a when dialog/interact is likely. Door exits can force a direction (e.g. down at front door).

4

LLM pick (optional)

Choose among candidates. No API key → use path[0] or direction_toward().

5

Execute

last_action = navigate_<dir> → apply_action presses button → stuck meter if position unchanged.

07 How you start a run

Different entry points skip different amounts of intro work.

--steps N

Cold boot (default)

Emulator bootstrap (title → bedroom) then graph bootstrap if needed → house exit → starter quest. Slowest path.

--start-bedroom

Bedroom fast-start

Skips graph bootstrap. Loads saves/bedroom_start.state if cached (no title); otherwise runs emulator bootstrap once and caches it.

--resume latest

Resume

Reloads agent checkpoint + latest emulator .state (headed). Continues where you left off — not a fresh start.

10 React loop — deep dive

Classic Perceive → Reason → Act → Reflect → Remember, implemented as a fixed LangGraph path every invoke. The agent never “thinks” in a free loop — it runs this pipeline exactly once per runner step.

Perceive

GameState from WRAM (before specialist)

supervisor_node reads state["game_state"] — parsed by GoldStateReader after the previous step's apply_action. Includes map_key, (x,y), in_text_box, script flags, battle state, event flags. No screen pixels.

Reason

Supervisor + one specialist

Supervisor picks exactly one node (bootstrap, navigator, interactor, waiter, battler, planner, idle). Specialist sets last_action and position_before_action but does not touch the emulator yet. Navigator/planner/battler may call the LLM here (network latency hits before any button press).

Act

apply_action_node → PyBoy

Executes last_action: press_button for navigate/bootstrap/interact, tick() for waiter/script waits, battle_decide tool for battles. Re-reads RAM into game_state. Updates stuck_count by comparing position before vs after. Phase hooks (on_map_change) fire on warps.

Reflect

critic_node

Checks last 3–5 history entries and stuck_count. If repeating the same action with stuck ≥ 3, or stuck ≥ 10 (env STUCK_THRESHOLD), sets should_replan = True and critic_verdict = "replan". Next step's supervisor will route to planner first.

Remember

memory_node → END or loop

Appends map to maps_visited, checks milestones, increments metrics.steps. If steps < run_max_steps (runner set this to current+1), routes back to supervisor within the same invoke — but runner only allows +1 step, so this branch returns END immediately. Checkpoint saved after invoke completes.

Why it feels slow when watching: On navigator/battler/planner steps, the LLM call happens in the Reason phase before Act. You see a multi-second pause, then one tile of movement. Bootstrap/interactor/waiter steps skip LLM entirely.

11 LLM decision making — deep dive

Three LLM calls in src/graph/llm.py. All use temperature 0, strict token matching, and heuristic fallback. Provider priority: OpenRouter → xAI → OpenAI.

Navigator llm_navigate()

Called every navigator_node step when candidates exist
System: You are the Navigator for Pokemon Gold. Pick one direction. Human: Map: Player's House 2F at (3,4) Subgoal: Leave player house Visited count: 12 Choose ONE direction from: a, right, down Reply with only the direction word. → matched token must be in candidates list
Fallback chain: LLM pick → else path[0] → else direction_toward(target). Candidates built from pathfinding + phase rules (e.g. insert a when dialog likely).

Planner llm_plan()

Only when supervisor routes to planner (stuck / replan) AND planner_allows_llm
System: You are the Planner for a Pokemon Gold autonomous agent. Human: Map: Route 29 (24:3) at (10,12) Party: 1, Badges: 0 Battle: False Current map only — subgoals must be achievable on Route 29 now. Reply with exactly 3 short subgoals, one per line, no numbering. → first 3 non-empty lines become subgoals
Fallback: Phase decompose_subgoals() (house/starter hardcoded lists) → generic explore strings. LLM disabled on Player's House 1F/2F always; Elm's lab & Mr. Pokémon house for planner.

Battler llm_battle()

Every battler_node step while in_battle
System: You are the Battler for Pokemon Gold. Human: Wild/trainer battle. Player HP 18/20, Enemy RATTATA HP 8/8. Can run: True Reply with one word: fight, run, switch, or item. → fight | run | switch | item
Fallback: run if HP < 20% and can_run; else fight. LLM output parsed via _match_token — substrings like "flight" won't match "fight".

Decision flow — navigator step (most common)

A

Phase provides target tile

e.g. house_exit.navigation_target() → stairs (7,0) on 2F

B

Pathfinding shrinks the search space

find_path → e.g. ["down", "down", "right"]. LLM never sees the full map.

C

Candidates = bounded menu

Up to 3 path steps + primary direction + optional a. LLM can only pick from this list.

D

LLM tie-break (optional)

~1 API call, ~1–5s latency. Picks e.g. down when both down and a are valid.

E

Single button press

last_action = "navigate_down" → apply_action → critic checks if tile actually changed.

What the LLM does NOT do

Choose the quest target (phase modules do) Read raw RAM or pixels Press buttons directly Run multiple actions per step Supervisor routing Dialog timing (interactor/waiter heuristics)

Stuck → replan feedback loop

When navigation fails repeatedly, stuck_count rises. Critic sets should_replan. Next step: supervisor → planner (may call LLM for new subgoals) → navigator again. replan_count indexes into the subgoal list so the active subgoal rotates on repeated replans. Interact steps reduce stuck count even without movement (dialog progress).

12 Specialist cheat sheet

bootstrap

A/B/Start through title, name entry, clock. Heuristic button schedule.

navigator

Pathfind toward phase target. Sets navigate_*.

interactor

Mash A/B through dialogs and indoor scripts.

waiter

Tick frames when script has joypad disabled. No buttons.

battler

LLM or heuristic fight/run during battles.

planner

Replan subgoals when stuck. LLM optional on outdoor maps.

idle

Phase complete — no further navigation input.

critic + memory

Always run after every action. Detect loops; record milestones.
